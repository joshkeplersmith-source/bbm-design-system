Agent Operating Instructions
BB-BRAND-PRO.html · Version 2.0 · Josh Smith + Claude Code shared operating contract. Read this section fully before building anything. All rules are active and non-negotiable.

Pre-Build Checklist

  • Confirm this file is the current brand source of truth
  • Page/deliverable base is N-100 #f1f0ee — not BU-700, not a gradient
  • Logo zone confirmed — what color is the surface the logo sits on?
  • Output type confirmed — route to correct workflow (Workflow section)
  • Print/poster → Illustrator MCP (MCP section). Not HTML
  • Document/report PDF → HTML + Chrome browser-print
  • Image source identified — generate, R2 reference, or brief Josh
  • R2 access tested — apply graceful degradation if blocked
  • Print only: does all copy fit within zone allocations at minimum text sizes? If not, edit copy to fit before building

Environment Routing — Brand Fidelity First

Before building any visual output, confirm the environment can render EB Garamond, Playfair Display, and Work Sans. If it cannot, route to HTML with Google Fonts via <link>. Never produce output with degraded fonts.

For print collateral, do not use HTML — route to Illustrator MCP (MCP section).

Image Requirement

Every visual deliverable must include at least one image. No purely text-forward layouts — ever.

  • Environment supports image generation → generate using photography brief
  • R2 accessible → suggest a reference from the photo library
  • Neither → pause and brief Josh (subject, mood, shot direction, 500T prompt)

While Building

  • Default to N-100. Never open a standalone deliverable on BU-700 or a gradient unless the brief calls for it
  • Apply the logo decision tree before placing any logo
  • Never use a color outside the six families
  • Never use a font outside the three families
  • Upright EB Garamond always leads — italic always follows, never opens
  • Reference color pairings by code (D1–D6, L1–L6)
  • Apply voice and vocabulary rules to all copy
  • Include textures only when R2 is accessible — never forced

Safe Baseline When Unsure

N-100 background · BU-700 section accent · EB Garamond heading (upright leads) · Work Sans 300 body · GL-300 on dark / GL-700 on light

Anti-Rationalization Rule

If you find yourself thinking a rule violation will look better, that reasoning is incorrect. The rules define what looks better for this brand. The designer knows what they want. Do not substitute agent aesthetic judgment for the rule system.

Creative Problem-Solving Fallback

Before diverging from any layout rule, exhaust these in-system options first:

  • Gradient tiles (neutral, female, male) for dimension on a light surface
  • NV-300 or NV-100 surface cards for hierarchy without dark backgrounds
  • L-bracket gold frame for featured or callout content
  • Texture on N-100 for depth without breaking the light base
  • Ghosted Playfair Display text for visual weight on N-100 sections
  • BU-700 for one interior section break only — not the document frame

Pre-Output Verification Gate — Run Before Presenting Any Deliverable

  1. What is the page/document base? It must be N-100.
  2. Are textures used anywhere? If yes — is that zone N-100? If not, remove the texture.
  3. Does any dark zone constitute the majority of the document surface? If yes, restructure.
  4. Is every color, font, and layout element traceable to a rule in the visual spec?
  5. (Print only) Does any text-bearing zone use a fixed height? Convert to min-height.

Print Copy Fitting Rule

If copy cannot fit within the page at minimum text sizes: (1) Rewrite the copy to fit — preserving meaning, voice, and all required information. (2) Complete the deliverable using the rewritten copy. (3) Surface the issue after output, showing exactly what was changed and why. The agent never shrinks text below minimum size. The agent never overflows the page. The agent never asks permission mid-build — it resolves the problem, delivers the file, and reports what it did.

Font environment note: The Claude Code / bash environment cannot reach Google Fonts CDN. Weasyprint falls back to system serif/sans fonts. Always use HTML → Chrome browser-print workflow for any output requiring correct font rendering.

Graceful Degradation

State 1 — R2 Accessible: Pull all logos and textures from asset library. Build normally.

State 2 — R2 Not Accessible: Colors, typography, spacing, voice, and layout rules remain fully available. Textures: Omit silently. No mention, no flag. Logo: Say: "I can't access the logo files in this environment. Drop the logo you need into this chat and I'll use it. BB-H-Logo-B.svg for light backgrounds, BB-H-Logo-W.svg for dark backgrounds." Wait for upload. Do not substitute a placeholder.

Infrastructure note: Neither raw.githubusercontent.com nor *.r2.dev resolve in Claude.ai chat. R2 URLs do resolve in Claude Code sessions and all browser-rendered outputs. Long-term fix: host assets at assets.bodybalancemedical.com.

Flagging Gaps

Flag specifically and wait for input. Never silently produce an off-brand result. Never guess at a brand color, logo file, or voice rule.

Typography Rules

DO

  • Upright EB Garamond always leads in headings — italic always follows, never opens
  • Use Playfair Display for subheadings, captions, buttons, UI labels, eyebrows, ghosted text
  • Use Work Sans 300 for all body copy; Work Sans 600 for bold emphasis sparingly
  • Ghosted Playfair Display italic at 8–12% opacity on N-100 sections only, once per page

DON'T

  • Open a heading with the italic span
  • Use EB Garamond outside headings and card titles
  • Use Work Sans for headings, subheadings, or UI labels
  • Use uppercase on EB Garamond
  • Mix heading fonts on a single surface

10 — Texture System

7 texture SVGs deployed via CSS ::before pseudo-element. NV-200 fill (#deddd3). 200vmax × 200vmax square, centered, rotated 90°. N-100 sections only — never on BU-700. One texture per section maximum. Never stack with logo mark watermark.
Correct CSS Implementation
.section-with-texture { position: relative; overflow: hidden; } .section-with-texture::before { content: ''; position: absolute; top: 50%; left: 50%; width: 200vmax; height: 200vmax; transform: translate(-50%, -50%) rotate(90deg); background-image: url('[R2-texture-url]'); background-size: cover; background-position: center; opacity: .22; pointer-events: none; z-index: 0; mix-blend-mode: multiply; } .section-with-texture > * { position: relative; z-index: 1; }

Section 16

Framing Principles

Six principles that govern how every piece of BBM content is framed. These are architecture rules — they apply before the first word is written.

Principle 01
Lead with the feeling, not the treatment.
Open with how the patient feels before treatment, not with the name of the service. "Tired but can't sleep" leads. "Hormone pellet therapy" follows. Never reverse this order in an ad or email subject line.
Principle 02
Restoration, not transformation.
BBM returns people to themselves — not a new, better, transformed version. "Feel like yourself again" is always correct. "Become your best self" is always wrong. The frame is return, not upgrade.
Principle 03
Medical authority, human delivery.
The content is clinically grounded — always. The delivery is warm, not academic. Write like a trusted clinician talking to a patient they know, not like a pharma insert or a lifestyle blog.
Principle 04
Candidacy, not urgency.
BBM does not pressure. CTAs invite qualification — "See if you're a candidate" — not manufactured urgency — "Limited spots!" or "Act now!" Scarcity messaging is banned unless it is literally true and has been confirmed by Mia or Teresa.
Principle 05
Educate before you sell.
Organic content is educational first. Myth-busting, explainers, and patient questions are the primary mode. The conversion happens at trust, not at pitch. Organic posts should almost never read as ads.
Principle 06
The two providers are the brand.
Mia and Teresa's complementary expertise is the core differentiator. Content should consistently reflect that this is a two-NP practice with overlapping specialties by design — not a solo provider with a colleague. Surface both names and credentials whenever the format allows.

Non-negotiable rule: Never rationalize exceptions. If a piece of content might violate a rule below, flag it and ask before submitting. Wrong is not a style judgment — it is a legal-exposure call.

RequiredBanned

Format Defaults

Format rules are not suggestions. Apply these before any output is considered complete.

Email
  • Subject line: 40 chars max (mobile preview cutoff)
  • Preview text: 85–100 chars — never blank
  • Single CTA per email — never two competing CTAs
  • CTA label must match Naming Conventions table
  • Em-dash rule: space–em-dash–space ( — ) not hyphen
  • No Oxford comma unless required for clarity
  • Sentence-case subject lines only — not Title Case
Meta / Paid Ads
  • Primary text: 125 chars before truncation on mobile
  • Headline: 40 chars max
  • Description: 30 chars max
  • Lead with symptom/feeling in primary text — never service name
  • Single CTA only — Meta allows one CTA button
  • LegitScript badge mention in GLP-1 / weight loss ads
  • No exclamation marks in headlines
FAQ Copy
  • Question: patient voice ("Does it hurt?" not "Is the procedure painful?")
  • Answer: clinician voice — warm, direct, no hedging filler
  • Max 3 sentences per FAQ answer — if longer, break into bullets
  • Never guarantee outcomes in FAQ answers
  • End with CTA only when contextually appropriate — not every FAQ
Punctuation
Punctuation & Typography
  • Em-dash: always with spaces — like this — never like-this
  • Ellipsis: three dots with spaces ( . . . ) in body; tight (…) in headlines only
  • Apostrophes: curly, never straight ('not ')
  • Quotation marks: curly doubles (" ") in body; avoid in headlines
  • Numbers: spell out one–nine; numerals for 10+; always numerals for measurements, ages, doses
  • Percent: 80% not 80 percent (numerals + symbol)
